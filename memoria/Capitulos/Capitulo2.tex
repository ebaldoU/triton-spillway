\chapter{Tecnología: TileDB}\label{tecnologia-tiledb}

Este capítulo describe el funcionamiento interno de TileDB: el modelo de datos sparse, la gestión de fragmentos y el mecanismo de compresión, con el objetivo de justificar las decisiones de diseño adoptadas en el capítulo siguiente.

\section{¿Qué es TileDB?}\label{que-es-tiledb}

TileDB es una base de datos orientada a arrays multidimensionales de código abierto. A diferencia de las bases de datos relacionales (tablas), TileDB organiza los datos en arrays N-dimensionales, lo que lo hace especialmente eficiente para datos científicos, rásteres geoespaciales y series temporales densas o dispersas.

TileDB no requiere servidor: los arrays se almacenan directamente en el sistema de ficheros local (o en la nube) como directorios con metadatos, fragmentos de datos y ficheros de índice. El acceso se realiza mediante la biblioteca Python (o C/C++/R) sin ningún servicio en ejecución. Esta arquitectura sin servidor reduce la complejidad operativa respecto a alternativas como SciDB o Rasdaman, que exigen un demonio en ejecución continua, lo que es especialmente relevante para despliegues en máquinas de uso compartido o con recursos limitados.

\section{Arrays sparse vs. densos}\label{arrays-sparse-vs.-densos}

TileDB soporta dos modos de almacenamiento:

\begin{itemize}
\item \textbf{Dense:} Almacena todos los valores del dominio, incluidos los nulos. Eficiente cuando el dominio está mayoritariamente relleno.
\item \textbf{Sparse:} Almacena únicamente las celdas no nulas junto con sus coordenadas. Eficiente cuando el dominio es mayoritariamente vacío.
\end{itemize}

En este proyecto se utiliza el modo disperso porque solo el $\sim$8,3 \% del dominio tiene agua. Almacenar el $\sim$92 \% restante (terreno seco) sería un desperdicio de espacio y tiempo de lectura. La Figura~\ref{fig:dense-vs-sparse} ilustra la diferencia entre ambos modos.

\section{Fragmentos y consolidación}\label{fragmentos-y-consolidacion}

Cada escritura en TileDB crea un fragmento independiente en disco. En este proyecto se realiza una escritura por paso temporal, lo que produce \textbf{20 fragmentos} (uno por cada intervalo de 6 horas). Mantenerlos separados tiene una ventaja importante: TileDB puede filtrar por la dimensión \texttt{time} leyendo solo los fragmentos relevantes, lo que acelera las consultas acotadas temporalmente.

La API de TileDB ofrece \texttt{tiledb.consolidate()} para fusionar todos los fragmentos en uno solo, y \texttt{tiledb.vacuum()} para eliminar los fragmentos anteriores. En este proyecto esta operación \textbf{no se aplica}: la consolidación agrupa celdas de distintos pasos temporales en los mismos tiles, lo que obliga a leer datos innecesarios en consultas que solo acceden a uno o pocos pasos (\textit{tile amplification}). Las pruebas realizadas durante el benchmark de configuración (véase \hyperref[anexo-b.-tablas-de-resultados-completas]{Anexo~B}, grupo G4) confirman que mantener un fragmento por paso temporal es la opción más eficiente para el filtrado por tiempo; la consolidación en un único fragmento penaliza las consultas multi-paso por \textit{tile amplification}.

\section{Compresión}\label{compresion}

Se utilizó el filtro ByteShuffle+ZSTD nivel 9 \cite{tiledb_compression_tutorial, tiledb_compression, tiledb_docs_compression}. Tras un \textit{benchmark} exhaustivo de 15 configuraciones (ZSTD puro niveles 1, 3, 5 y 9, ByteShuffle+ZSTD y BitShuffle+ZSTD en los mismos niveles, y DeltaFilter+ZSTD inaplicable en float32) se concluyó que las diferencias de velocidad de lectura entre configuraciones son menores que el ruido de la caché del sistema operativo. El criterio determinista es el tamaño en disco: ByteShuffle+ZSTD-9 ofrece el menor tamaño (860 MB/paso en la región de prueba) gracias al reordenamiento de bytes de los float32, que agrupa exponentes y mantisas similares antes de comprimir. Reduce el almacenamiento un 8,9 \% adicional respecto a ZSTD-1 sin penalización apreciable en lectura.

La sección~\ref{diseno-del-sistema} presenta el diseño del array TileDB y el proceso ETL de carga de datos. El \hyperref[anexo-d.-proceso-de-ingesta-etl-detallado]{Anexo~D} describe el proceso de ingesta con mayor nivel de detalle.

\chapter{Diseño del Sistema}\label{diseno-del-sistema}

El sistema está compuesto por cuatro capas articuladas en torno al array TileDB sparse como componente central de almacenamiento, tal y como muestra la Figura~\ref{fig:arquitectura}:

\begin{enumerate}
\item \textbf{Ingesta (ETL):} El módulo de ingesta lee los ficheros GeoTIFF generados por Triton, aplica el filtro de celdas húmedas (H $\geq$ 0,01 m) y escribe los datos en TileDB con filtro ByteShuffle+ZSTD-9. La lectura se realiza por franjas de 5 000 filas para limitar el uso de memoria.
\item \textbf{Almacenamiento:} Array TileDB sparse con dimensiones (time, row, col) y atributos (H, QX, QY, MH). Un fragmento por paso temporal facilita el filtrado temporal eficiente.
\item \textbf{Análisis:} El motor de consultas proporciona 20 funciones que operan directamente sobre el array TileDB sin exportación previa.
\item \textbf{Visualización y exportación:} Un visualizador interactivo, un exportador a GeoTIFF para interoperabilidad SIG y la aplicación web Streamlit con mapas interactivos, comparativa multi-escenario y panel de ayuda.
\end{enumerate}

\begin{enumerate}
\item Abre los 4 ficheros GeoTIFF (H, QX, QY, MH) correspondientes con rasterio.
\item Lee el ráster en franjas de 5 000 filas (CHUNK\_ROWS) usando lectura por ventana (windowed read) para evitar cargar 3 GB/variable en RAM.
\item Aplica la máscara de celdas mojadas: H $\geq$ 0,01 m.
\item Acumula las coordenadas y valores de las celdas mojadas en listas.
\item Realiza una única escritura en TileDB por paso temporal $\rightarrow$ 1 fragmento por paso.
\end{enumerate}

La lectura paralela de las 4 bandas se realiza con \texttt{ThreadPoolExecutor(max\_workers=4)}, reduciendo el tiempo de I/O.

Memoria pico estimada: 4 bandas $\times$ 5 000 filas $\times$ 23 040 cols $\times$ 4 B $\approx$ 1,8 GB.

\section{Gestión de múltiples escenarios}\label{gestion-de-multiples-escenarios-argumento---dataset}

El sistema está diseñado para procesar y almacenar múltiples escenarios de simulación de forma independiente. Cada escenario ocupa su propio array TileDB en disco y se identifica mediante un alias corto (datos1, datos2, \ldots) que \texttt{config.py} traduce al nombre real del directorio de salida de Triton. De este modo el resto del código (ETL, consultas, aplicación web) opera siempre con el alias y no depende de los nombres largos de los ficheros fuente.

El módulo de ingesta acepta el argumento \texttt{-\/-dataset} para seleccionar el escenario a procesar. Dos variables de entorno controlan las rutas:

\begin{itemize}
\item \texttt{TRITON\_BASE\_URI}: directorio donde se crearán los arrays TileDB, bajo el subdirectorio \texttt{triton\_results/}.
\item \texttt{TRITON\_GTIFF\_DIR}: directorio raíz donde están las carpetas de GeoTIFF fuente (\texttt{datos1/}, \texttt{datos2/}, etc.), cada una con los 80 ficheros del escenario correspondiente.
\end{itemize}

Un ejemplo de invocación para el escenario \texttt{datos1} en local:

\begin{verbatim}
TRITON_BASE_URI=/ruta/datos/triton_results \
TRITON_GTIFF_DIR=/ruta/datos \
  python geotiff_to_tiledb_sparse.py --dataset datos1
\end{verbatim}

La estructura de directorios esperada es:

\begin{verbatim}
/ruta/datos/
  triton_results/        <- TRITON_BASE_URI
    output_10_.../       <- arrays TileDB generados
  datos1/                <- TRITON_GTIFF_DIR/datos1
    H_06_00.tif
    QX_06_00.tif  ...
\end{verbatim}

La aplicación web descubre automáticamente los \textit{datasets} disponibles inspeccionando el directorio \texttt{triton\_results/} apuntado por \texttt{TRITON\_BASE\_URI}: cualquier subdirectorio que contenga un array TileDB válido aparece en el selector de la interfaz. La fecha de inicio del escenario se extrae del nombre del directorio fuente (formato \texttt{..\_YYYYMMDD\_YYYYMMDD}), sin necesidad de lectura adicional del array. Añadir un nuevo escenario consiste en ejecutar el ETL con el alias correspondiente; no es necesario modificar ningún fichero de configuración adicional.
